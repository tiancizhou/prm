# Bug 版本字段移除设计

## 背景

当前 Bug 列表页与详情页的“解决版本 / 影响版本”字段并未接入真实版本数据。

- 前端选项为写死值
- 后端 Bug 接口与数据模型未承载这两个字段
- 重开时“影响版本”仅拼接到备注，不落库

这会让用户误以为系统具备可维护的版本设置能力，实际却没有形成闭环。

## 目标

移除 Bug 处理流程中的“解决版本 / 影响版本”两个占位字段，降低误导和无效输入。

## 范围

- 移除 `frontend/src/views/bug/BugPage.vue` 中：
  - 解决弹窗的“解决版本”字段
  - 重开弹窗的“影响版本”字段
- 移除 `frontend/src/views/bug/BugDetailPage.vue` 中对应两处字段
- 清理这两个字段关联的表单状态、校验规则与备注拼接逻辑

## 不做

- 不新增版本管理页面
- 不改后端 Bug 数据模型
- 不把 Bug 与 `pm_release` 建立正式关联

## 方案对比

### 方案 A：仅取消必填，保留字段

优点：改动最小。

缺点：字段仍是假数据，继续误导用户。

### 方案 B：直接移除两个字段

优点：与当前系统能力一致，界面更干净，避免无效输入。

缺点：未来若要接入版本管理，需要重新设计字段与数据流。

### 方案 C：立即接通 Release 模块

优点：语义完整。

缺点：超出本次范围，需要补路由、菜单、API、Bug 模型与迁移。

## 决策

采用方案 B：直接移除两个字段。

## 影响

- 解决 Bug 时仍保留解决方式、指派人与备注
- 重开 Bug 时仍保留指派人与备注
- 评论中不再写入“解决版本 / 影响版本”文本

## 验证

- 页面能够正常编译
- 两个页面不再出现相关字段
- 解决 / 重开流程仍能正常提交
