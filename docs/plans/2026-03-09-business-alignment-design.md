# PRM 业务流程收敛整改设计

**日期：** 2026-03-09  
**目标：** 在不改表、不加迁移、不新增页面的前提下，将当前 PRM 系统从“半成品混合态”收敛为“规则一致、口径一致、用户可解释”的可用态。  
**约束：** 第一阶段不改数据库结构，不新增持久化字段，不新增页面与导航入口。

---

## 1. 背景

当前项目的核心问题不是单点 Bug，而是业务规则在多个模块之间存在系统性不一致：

- 权限模型混用“项目成员关系”和“全局角色”两套口径
- 前后端对状态机的理解不一致，页面动作与后端真实流转脱节
- 看板和项目概览存在伪数据或口径漂移
- 多个跨模块能力只实现了“表结构/接口/占位 UI”的一部分，未形成闭环
- 设计文档描述的是目标态，但实现已收敛为更简化的现状态，二者存在偏差

如果继续在当前基础上叠加页面与功能，会把不一致继续扩散到更多模块。

---

## 2. 总体策略

### 2.1 阶段划分

本次采用两阶段策略：

- **阶段一：收敛现有能力**
  - 不改表、不加迁移、不新增页面
  - 统一权限、状态机、页面动作、统计口径
  - 下线或降级半成品能力，避免误导用户
- **阶段二：补齐真正闭环**
  - 在阶段一规则稳定后，再决定是否正式启用 Release、Sprint 规划项、需求-Bug 关联、真实趋势图等跨模块能力

### 2.2 第一阶段设计原则

- 宁可下线半成品能力，也不保留误导性能力
- 同一业务在前端、后端、统计口径上只允许存在一套主规则
- 历史状态和旧字段可兼容展示，但不能继续作为现行主流程扩散
- 所有用户可见数据都必须来源于真实接口或真实持久化数据

---

## 3. 第一阶段目标

第一阶段完成后，系统应满足以下标准：

1. 每个已开放模块都能回答清楚：
   - 谁能看
   - 谁能改
   - 状态怎么流转
2. 前端展示的按钮、动作、标签与后端真实业务能力一致
3. 页面展示的数据不再包含随机模拟或隐性伪造
4. 文档中明确区分“当前已启用能力”和“后续阶段目标能力”

---

## 4. 第一阶段范围

### 4.1 纳入范围

- 权限统一：`task`、`sprint`、`release`
- 状态机收敛：`bug`、`task`、`requirement`
- 页面动作和文案清理
- 项目概览、Dashboard 统计口径修正
- 半成品能力降级说明与设计文档同步

### 4.2 明确不做

- 不新增 `Release` 页面、路由、菜单
- 不启用 `pm_release_item`
- 不启用 `pm_sprint_item`
- 不启用 `pm_requirement_link` 的正式双向关联能力
- 不恢复完整需求评审流
- 不增加需要新字段承载的能力

---

## 5. 第一阶段整改包

### 5.1 整改包一：权限统一

**目标：** 统一项目域模块的权限模型。

**现状：**

- `project`、`requirement`、`bug` 已经采用“项目成员关系 + 项目管理员”的权限模型
- `task` 使用 `SecurityUtil.isManager()`，依赖全局角色 `PROJECT_ADMIN`
- `sprint`、`release` 服务层几乎没有项目级权限边界

**设计决策：**

- 第一阶段以 `project / requirement / bug` 当前的项目成员制为基准
- `task / sprint / release` 全部向项目成员模型对齐
- 超级管理员保留全局放行
- 项目普通成员仅可查看自己有权数据，项目管理员可查看和操作项目级数据

**预期结果：**

- 不同模块不会再因权限模型不同而给出相互矛盾的结果

### 5.2 整改包二：状态机收敛

**目标：** 每个模块只保留一套主状态语义。

#### Bug

- 主状态收敛为：`ACTIVE / RESOLVED / CLOSED`
- `NEW / CONFIRMED / ASSIGNED / VERIFIED` 仅作为历史兼容展示，不再作为主流程状态扩散
- 页面上的“验证”动作不能继续用“关闭”来冒充

#### Task

- 以后端现有五态为准：`TODO / IN_PROGRESS / PENDING_REVIEW / DONE / CLOSED`
- 前端动作必须按当前状态收敛，不允许继续展示“全部状态可选”

#### Requirement

- 第一阶段确认主流程为：`DRAFT / IN_PROGRESS / DONE`
- `REVIEWING / APPROVED / CLOSED` 仅作为兼容态，不作为当前流程的核心路径

**预期结果：**

- 前端按钮、后端流转校验、统计口径三边一致

### 5.3 整改包三：页面动作与文案清理

**目标：** 去掉误导性的页面表达。

**设计决策：**

- 用户只能看到真实可走通的动作
- 名称必须与实际结果一致
- 未启用的业务流不再通过按钮、标签、下拉文案继续暗示

**预期结果：**

- 页面上的每个动作都具备真实业务含义

### 5.4 整改包四：统计真实性修正

**目标：** 所有用户可见统计与图表都来自真实数据。

**现状问题：**

- `ProjectOverview` 使用 `Math.random()` 生成趋势图
- “项目概览”在非项目管理员场景会被后端自动降级为“个人视角”，但页面文案仍是项目概览
- Bug 未关闭口径还在兼容 `VERIFIED`

**设计决策：**

- 第一阶段优先真实，必要时宁可降级展示
- 先下线伪趋势图，只保留真实 KPI、真实活动流、真实列表指标
- 项目概览与 Dashboard 必须明确口径，避免“名字是项目概览，内容是个人概览”

### 5.5 整改包五：半成品能力降级

**目标：** 对未形成闭环的能力做边界澄清。

**第一阶段判定为未启用闭环的能力：**

- `release`：当前仅视为“版本记录能力”，不是完整发布闭环
- `pm_release_item`：未启用
- `pm_sprint_item`：未启用
- `pm_requirement_link`：未启用正式双向关联，仅保留现有转需求行为
- `RequirementReview`：接口与实体保留，但不视为当前主流程能力
- `pm_operation_log`：第一阶段已为核心写操作接通控制器注解审计；当前记录粒度以 `module/action/bizType/bizId/afterData` 为主，如需字段级 before/after 差异审计可在后续阶段增强

**设计决策：**

- 第一阶段不强行补齐这些能力
- 只通过文档、文案、动作控制把边界说清楚

---

## 6. 第一阶段整改顺序

为降低返工，执行顺序固定为：

1. 统一权限
2. 统一状态机
3. 清理页面动作与文案
4. 修正统计真实性
5. 同步文档与半成品能力边界

原因：

- 权限与状态机是页面动作和统计口径的共同基础
- 若先改页面，后改规则，会导致明显返工
- 若先改图表和统计，但状态机未统一，数字口径仍然不稳定

---

## 7. 风险与应对

### 风险 1：权限收紧后，部分用户突然失去访问/操作能力

**应对：**

- 以现有 `project / requirement / bug` 的权限规则为对齐基准
- 为 `task / sprint / release` 补权限测试，覆盖超级管理员、项目管理员、普通成员、非成员

### 风险 2：历史状态与现行主状态并存

**应对：**

- 第一阶段不迁移历史数据
- 保留兼容展示，但限制新操作只进入主状态流

### 风险 3：用户感觉功能减少

**应对：**

- 通过文案明确“当前流程”
- 删掉的是伪能力，不是真能力

### 风险 4：统计图形暂时简化

**应对：**

- 优先保真，再补美观
- 第二阶段再引入真实趋势接口

---

## 8. 验证策略

### 8.1 权限验证

- 为 `task / sprint / release` 补服务层权限测试
- 对照现有 `project / requirement / bug` 的测试模式

### 8.2 状态验证

- 验证每个状态机的合法流转与非法流转
- 验证前端展示的动作集合与后端允许流转一致

### 8.3 页面验证

- 核对每个主页面的按钮、标签、详情状态展示与实际业务一致

### 8.4 统计验证

- `Dashboard`、`ProjectOverview`、模块列表页的统计口径要能互相对照

### 8.5 真实性验证

- 禁止继续存在：
  - 随机趋势图
  - 无后端承接的占位字段
  - 暗示未启用业务流的按钮或入口

---

## 9. 第二阶段展望

第二阶段再决定是否正式启用以下能力：

- `Release` 页面与发布闭环
- `pm_release_item` 发布项关联
- `pm_sprint_item` 迭代规划项
- `pm_requirement_link` 需求与 Bug 正式关联
- Dashboard / ProjectOverview 真实趋势接口
- 需求评审流正式恢复

这些工作都需要接受 schema 变更与迁移成本，因此不纳入第一阶段。

---

## 10. 结论

第一阶段的本质不是“继续堆功能”，而是“先把现有能力说清楚、做真实、做一致”。

只有在权限、状态机、页面动作、统计口径都稳定之后，第二阶段去启用跨模块闭环能力才不会继续扩大系统复杂度和返工成本。
