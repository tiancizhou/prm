# PRM Admin UI Alignment Design

**Date:** 2026-03-16

## Background

当前 `后台` 域已经从旧的占位页演进为真实的人员管理域，包含：

- `后台 > 部门`
- `后台 > 用户`
- `后台 > 权限`

但这三页的视觉语言仍与当前 `组织` 域存在明显代差：

- `后台 > 部门` 已经部分接近 `组织 > 公司` 的“左侧结构区 + 右侧双卡”模式，但卡片节奏、侧栏样式、辅助文案层级仍未完全统一。
- `后台 > 用户` 与 `后台 > 权限` 仍保留更早一代后台工作区风格，工具栏、提示条、表格面板、分页和操作按钮与 `组织 > 团队` 的样式节奏不一致。
- 壳层层面还存在两个体验裂缝：
  - 左侧一级导航只在 `/admin/departments` 时稳定高亮“后台”，进入 `/admin/users` 与 `/admin/permissions` 时容易失去一级高亮。
  - 后台子页缺少精确的面包屑标签，导致进入后台页时顶部只剩“首页”或语义过粗。

用户本轮明确要求是：

1. **优化后台页面**
2. **与组织页面的 UI 保持一致**
3. **保持现有信息架构与业务交互不变**
4. **顺手修正壳层小不一致**

## Confirmed Decisions

- 本轮采用 **视觉一致**，不做后台信息架构重排。
- 页面级范围采用 **C**：
  - 优化 `后台 > 部门 / 用户 / 权限` 三个内容页
  - 同步修正顶部域导航细节
  - 同步修正左侧菜单高亮与面包屑等壳层小问题
- `后台 > 部门` 保留当前“左树 + 右双卡”结构。
- `后台 > 用户` 与 `后台 > 权限` 保留当前后台工作台式结构，不迁移为组织页同构页面。
- 本轮不修改后台接口协议、权限模型与后端逻辑。

## Goals

1. 让后台三页在视觉上与 `组织 > 团队`、`组织 > 公司` 属于同一代产品界面。
2. 保持后台现有业务交互、信息架构和高频操作路径不变。
3. 修复后台域在一级导航高亮与面包屑语义上的体验断层。
4. 让后续页面继续统一时，可以复用已有 `surface-card`、`workspace-*`、`page-header` 这套样式语言，而不是再造新体系。

## Non-Goals

- 不把后台三页重构成与组织页完全相同的页面结构。
- 不新增后台模块、入口或业务流程。
- 不修改后台域路由分组结构。
- 不修改后端权限与接口返回结构。
- 不顺手重做组织页，只把组织页作为对齐基准。

## Alignment Baseline

本轮以以下两个页面作为主要视觉对齐基准：

### `组织 > 团队`

提供以下视觉语义：

- 标准页头：标题、副标题、右侧主操作
- 单主卡容器：筛选区、表格区、分页区形成一体化节奏
- 更克制的工具栏与间距体系
- 表格容器、分页留白、链接式主文本的统一语气

### `组织 > 公司`

提供以下视觉语义：

- 左侧结构面板 + 右侧双卡的布局节奏
- 面板标题、路径文本、辅助说明的层级关系
- 侧栏卡、树面板卡、表单卡、子项维护卡的统一质感
- 未保存提示、底部操作区、分隔线的风格规范

## Page-Level Design

### `后台 > 部门`

#### Keep

- 保留左侧部门树。
- 保留右侧“当前节点卡 + 直属子部门卡”的双卡结构。
- 保留公司节点与部门节点的不同处理逻辑。
- 保留未保存变更提示与离开前确认。

#### Align

- 左侧树面板对齐 `组织 > 公司` 的侧栏卡：
  - 宽度、内边距、圆角、边框、阴影统一
  - 标题区与操作区的节奏统一
  - 树节点上下留白统一
- 右侧双卡对齐组织页卡片语言：
  - 路径文本降级为次要信息
  - 面板标题与按钮区强化主层级
  - 辅助说明文案使用统一的 helper text 节奏
  - 底部操作区与卡片内分隔线统一
- 输入区与按钮区维持当前交互密度，但外观向 `组织 > 公司` 收敛。

#### Expected Result

功能不变，但从视觉上明确成为“组织公司页的后台管理变体”，不再像独立旧系统。

### `后台 > 用户`

#### Keep

- 保留顶部筛选、搜索、主操作。
- 保留左侧部门树 + 右侧用户表格。
- 保留弹窗与用户项目范围抽屉。
- 保留图标型行操作与现有字段结构。

#### Align

- 顶部工具栏对齐 `组织 > 团队`：
  - 筛选按钮、搜索框、主按钮落在同一视觉基线
  - 搜索框宽度与按钮尺寸统一
  - 次要按钮与主按钮层级统一
- 规则提示区改为与组织页一致的浅底提示条风格。
- 左侧树卡与右侧表格卡统一为同一套 `surface-card` 观感。
- 表格区与分页区对齐 `组织 > 团队` 的密度、留白和边界处理。
- 图标操作区保留，但 hover、禁用态、间距与工具提示风格收敛。

#### Expected Result

后台用户管理仍然保持专业工作台属性，但界面不再与组织团队页割裂。

### `后台 > 权限`

#### Keep

- 保留顶部操作按钮区。
- 保留说明提示区。
- 保留角色分组表。
- 保留详情、成员、编辑、分配权限等弹窗结构。
- 保留“左角色列表 + 右模块权限”的授权弹窗布局。

#### Align

- 顶部操作区与 `组织 > 团队` 的动作栏统一节奏。
- 提示条统一为组织页 helper banner 风格。
- 角色分组表容器与分页/留白规则统一到组织团队页。
- 授权弹窗内部的角色列表、模块卡片、勾选项卡片统一卡片边距、圆角和标题层级。

#### Expected Result

权限页仍然保留其后台专业度，但在视觉层面显著回归到当前产品体系。

## Shell-Level Design

### Left Global Navigation

- `/admin/departments`
- `/admin/users`
- `/admin/permissions`

以上三个后台子路由都应该高亮左侧一级导航中的“后台”，而不是只有默认页高亮。

### Breadcrumbs

补齐以下后台子页语义标签：

- `AdminDepartments`
- `AdminUsers`
- `AdminPermissions`

这样用户进入后台任意子页时，都能看到具体面包屑而不是模糊域标签。

### Domain Top Navigation

- 不改变交互模型。
- 只微调标题区、页签区、激活态与整体间距节奏，使后台域与组织域显得来自同一套子壳层组件。

## Implementation Boundaries

### Expected File Scope

- `frontend/src/views/admin/AdminDepartmentPage.vue`
- `frontend/src/views/admin/AdminUsersPage.vue`
- `frontend/src/views/admin/AdminPermissionsPage.vue`
- `frontend/src/layouts/MainLayout.vue`
- `frontend/src/constants/layout.ts`
- `frontend/src/layouts/DomainTopNavLayout.vue`（仅在需要时轻量调整）

### Avoid Touching

- `frontend/src/api/adminDepartment.ts`
- `frontend/src/api/adminRole.ts`
- `frontend/src/api/adminPermission.ts`
- `backend/**`

## Validation

### Static Verification

- `npm.cmd exec -- vue-tsc -b`
- `npm.cmd run build`

### Manual Verification

登录后分别检查：

- `/admin/departments`
- `/admin/users`
- `/admin/permissions`
- `/organization/team`
- `/organization/company`

重点确认：

- 标题、副标题、卡片与间距是否处于同一视觉代际
- 工具栏、提示条、表格、分页、侧栏是否风格统一
- 左侧“后台”一级菜单在三个后台子页都保持高亮
- 面包屑是否显示具体后台子页标签
- 域顶部导航是否与组织域保持一致的节奏与激活态表现

## Expected Outcome

完成后，用户切换 `组织` 与 `后台` 域时，不会再产生“像两个产品拼接在一起”的观感：

- `后台 > 部门` 成为组织公司页风格的后台延展页
- `后台 > 用户` 与 `后台 > 权限` 回归当前产品的卡片式工作台节奏
- 壳层导航与面包屑的语义更稳定、更完整

最终结果不是“后台变成组织页”，而是“后台与组织页终于属于同一套 UI 体系”。
