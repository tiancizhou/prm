# Frontend Role Scope Wording Design

**Date:** 2026-03-13

## Background

当前系统已经确认采用单一角色模型：

- `系统角色` 决定功能权限
- `项目成员关系` 决定业务数据范围

因此，前端不应继续向用户传递“系统角色一套、项目角色又一套”的认知。

## Confirmed Decisions

本次设计依据已确认的用户输入：

- 前端最终采用**方案 A**：彻底去掉“项目角色”展示，只保留`系统角色` + `是否已加入项目`
- 后台和项目页都需要明确提示：`系统角色决定权限，加入项目后才可查看该项目业务数据`
- 整个系统坚持“一切从简”，不再为项目域保留第二套角色概念
- 本轮只做前端表达、信息结构与文案收敛，不新增新的权限模型

## Core Rule

前端统一使用以下产品规则：

1. `系统角色 = 权限能力`
2. `加入项目 = 数据范围`

所有页面的字段、标签、提示文案、空态说明，都应围绕这两句展开。

## Scope

### In Scope

- 项目成员页的表头、说明文案、空态表达统一
- 后台用户页的字段结构、详情信息、列表表达统一
- 后台权限页的说明文案与角色说明统一
- 相关常量文案统一，去掉“项目角色”含义
- 相关弹窗与表单去掉“项目角色选择”语义

### Out of Scope

- 后端权限模型再调整
- 新增角色体系或新的权限组模型
- 新增前端测试框架
- 大规模视觉重做

## UX Strategy

### 1) Project Members Page

目标：让用户明确这是“项目范围维护页”，不是“项目角色配置页”。

设计要点：

- 页面顶部增加规则提示：`系统角色决定操作权限；成员加入当前项目后，才能查看当前项目数据`
- 列表只展示：姓名、部门、职位、系统角色、加入时间、操作
- 移除任何“项目角色”字段或暗示性文案
- 添加成员动作只表示“把某人加入当前项目”，不提供项目角色选择

### 2) Admin Users Page

目标：让后台用户页清晰表达“角色能力”与“项目范围”的区别。

设计要点：

- 列表优先展示：姓名、部门、职位、系统角色、已加入项目数、状态、操作
- 将与本轮目标无关的弱价值字段降级或移除
- 在用户详情或侧边信息区增加`项目范围`表达，用于展示已加入项目或加入数量
- 所有角色标签都明确表示为`系统角色`

### 3) Admin Permissions Page

目标：让权限页只承担“系统角色权限配置”的职责。

设计要点：

- 页面顶部增加说明：`系统角色控制功能权限；是否加入项目决定业务数据可见范围`
- 所有“角色组 / 成员 / 模块权限”的说明都避免暗示存在独立项目角色
- 成员列表中的角色信息统一标注为系统角色

## Copy Rules

### Allowed Phrases

- `系统角色`
- `项目加入状态`
- `已加入项目`
- `项目范围`
- `加入项目后才可查看该项目业务数据`

### Disallowed Phrases

- `项目角色`
- `项目内角色`
- `成员角色`
- `项目权限角色`

## Empty State and Permission Messaging

所有“看不到数据”的场景，优先区分以下两类：

- 未加入项目：提示当前未被纳入该项目数据范围
- 无操作权限：提示当前系统角色不具备对应操作能力

避免继续使用会混淆语义的泛化提示，例如只写“权限不足”却不说明是角色问题还是范围问题。

## Success Criteria

- 前端不再把“项目角色”作为有效权限概念展示给用户
- 项目成员页只表达“谁加入了当前项目，以及他的系统角色”
- 后台用户页能清晰表达“系统角色 + 项目范围”
- 权限页只表达系统角色的权限，不再暗示存在独立项目角色
- 系统内关于角色与范围的文案保持一致
